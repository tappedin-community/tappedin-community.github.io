Tapped In 2: Course Manager

Functional Specification

Version 5.0

 

1. Overview

 

      This document describes the functional specifications for the Course Manager (1) for Tapped In Version 2.0.  The core implementation of the course management system is licensed from Prometheus, a third party.  The Prometheus source code will have to be modified and appended to accommodate 1) the features not already provided by Prometheus, 2) the proposed interface with Tapped In, and in later versions, possibly 3) integration with other course management systems such as Blackboard and Yahoo!.  An open dialogue with the writers of Prometheus will maintained such that whenever possible, large changes and additions to the Prometheus source code base can be implemented by Prometheus rather than the Tapped In team.

      The Course Manager is completely integrated into the Tapped In community and interface.  The course management functionality primarily appears in the Tapped In content area (see Integration into Tapped In below) when a user is in the Tapped In room for that course.  This is referred to as the “course room” (1) throughout the document.  Only Tapped In members that are enrolled as teachers, students, etc. in the course can access the course room.  The processes of finding, enrolling in and creating courses is also specified in this document.

      After laying out a few assumptions, the specification begins with descriptions of users, course attributes, an overview of course features, an explanation of how users find, enroll in, and create courses, then walks through the proposed interface organization for a course room, describing where and how users would access various functions and execute tasks.

2. Assumptions

 

      Tapped In has a number of features that will be applicable to the course manager.  These include but may not be necessarily limited to a chat facility, private messaging, and a means to share files and links in Tapped In.  The course manager specifications for some portions may allude to designs for other Tapped In features and be less specific than for functionality only described here.

      There is little distinction in the specification between functionality and interfaces of the course manager that are provided by Prometheus and those not provided by Prometheus.  The Prometheus code will be leveraged to meet the requirements of the specification as much as possible, with the remaining functionality to be either implemented by Prometheus, independently implemented by modifying Prometheus source code or pre-written open source code modules (e.g. a chat facility) to be integrated into Tapped In.  In all likelihood, several features provided by Prometheus will be unused.  It is worth noting that as more modification and in-house development is required, the implementation process will lengthen.

      There are some features/issues that are not yet addressed in this specification.  The following are some pending questions:

§         How does one remove oneself from a course?

§         Should ownership of a course (i.e. being the primary instructor) be transferable?

§         What happens when a course is given repeatedly and the semester begins again?  What parts of the interface will that affect? (e.g. Gradebook, Settings, Calendar, etc.)  Is there an easy way to change the course date, meeting times/days (once created), etc.?

§         Tracking of course specific usage data

 

3. Users

 

      There are primarily two types of users of a course management system: teachers and students. (1)  Accordingly, there are two basic interfaces (teacher and student) which are very similar for the most part. (1)  The most pronounced difference between the two interfaces is the ability of the user viewing the teacher interface to edit content, use the gradebook and access other teacher-specific utilities.  It is important to keep in mind that the user roles described here are in context with the course in question, and do not serve as Tapped In user roles at large.  For example, a teacher in one course might be a student in another.

      Prometheus provides four levels of course level access control (see excerpt below).  Not unexpectedly, a user at the faculty level would see the teacher interface and one at the student level would see the student interface.  The teaching assistant level (2) would view a version of the teacher interface with unusable functions not appearing in the interface and similarly, the shadow level (3) would view a version of the student interface with unusable functions not appearing.  Tapped In is room based, and each room has a single owner, which in this case will be an faculty level user.  This instructor is the primary instructor. (1)  Even if a fellow instructor or teaching assistant level user has full edit access permissions (as described in Settings below) only the primary instructor can delete the course.  All instructors always have full edit access.

4.2. Privacy Settings

 

      A course is specified as either public, private, (1) or public to tenant members (2).  Designating a course as public means that any Tapped In member can apply to enroll and that the course appears in the tenant course directory and the Tapped In course directory.  A private course does not appear in any directories whatsoever and users can enroll by invitation only.  Finally, a public to tenants course appears in the tenant and Tapped In directories only for members of the tenant organization with which the course is associated.  For example an NCREL public to tenants course would appear in the directories of an NCREL Tapped In user but not in those of a Pepperdine user.  Likewise, only members of the appropriate tenant organization can apply to enroll to a public to tenants course.  In short, a public to tenants course looks public to a tenant member and private to a non-tenant member.

      If a course does not appear in a user’s directories, then that course never appears in the search results for that user in any case (enrolled or not, any privacy setting).  Any member can be invited by an instructor to join a course, regardless of privacy settings.  None of the three settings allows users to enroll in courses without instructor approval (i.e. even public courses enrollment applications require instructor approval although not an invitation).  The instructor specifies the privacy setting upon course creation.  An instructor can make a course more public (e.g. from private to public to tenants, or public to tenants to public), but not more private (e.g. not from public to private).

      Once a user is enrolled, that course appears in the Me course directory, the Tapped In directory, and if applicable (i.e. the user is a member of the course tenant organization), the tenant directory, regardless of the privacy setting of the course.  Similarly, an appropriate search will always return courses that the user is currently enrolled in, regardless of the privacy setting of the course.

5. Outside the Course Room

 

      Course management functionality is encapsulated in a course room, but Tapped In users must have a way to access and create course rooms. The primary methods of doing this are by using the directories at the top of the Tapped In interface.  These are described below after which scenarios describe the processes of creating a course, enrolling students, and teaching a course (teacher/teaching assistant) as well as choosing, enrolling in, and taking a course (student).  Activities and tasks performed inside the course room are described in the Course Room Interface Organization section.

5.2 Course Information Page

5.3. Tasks Outside the Course Room

 

5.3.1. Finding a Course (Not Enrolled) - All Roles

 

      There are several ways for a user to find an existing course they are not enrolled in.  The two guaranteed ways are to either search for the course using the search function or to look in the directories at the top of the Tapped In interface.  However, courses will only appear in either of these places if the course is public to the user (see Privacy Settings).

      Other places a user might find a course is to try and take a passageway from another room to that course room.  Presumably, tenant organizations will, as a group, have a group room which would probably have a passageway or a link to tenant courses.

      Any of these options would result in getting the information screen for that course because the user is not yet enrolled.

 

5.3.2. Finding a Course (Enrolled) - All Roles

 

      A user that is enrolled can get to a course room by using any of the methods that are listed for finding a course in which the user is not enrolled.  In addition, however, the course appears in the Me directory course section.  Once enrolled, a user can also designate the course room as a favorite place or build a passageway from the user’s home office to the course room for easy access.

 

5.3.3. Enrolling in a Course (All Roles but Primary Instructor)

 

      To enroll in a course as a student, the user must first find the course (see Finding_a Course (Not_Enrolled) – All Roles), and, having arrived at the information page for that course, click on the link that says “Apply to Enroll as a Student”.  To enroll as an instructor, teaching assistant, or shadow user, the user should click the appropriate “Apply To Enroll” link.  This will send a private message to the relevant instructors and teaching assistants of the class describing the enrollment request (username, user type). (1) Pending enrollment requests are also retained in the Student Management course settings for later review by instructors and teaching assistants. (2) An instructor or teaching assistant will then either approve or reject the user’s enrollment and the user will be notified by private message of the result.  After that, if the user is accepted, the course will appear in the Me directory and clicking on the course name will take the user to the course room rather than the course information page.  For specific information on teaching assistant privileges in regards to enrollment, see the Settings section.

 

5.3.4. Creating a Course (Primary Instructor)

 

      A tenant member can create a course my going to the Courses portion of any of the three directories and clicking on “Create a Course”.  This will take the user to an edit screen where he or she can specify the following: Course Department/Number/Section/Title (together forms the course name), Privacy Setting, Course Dates, Meeting Days & Times, and Keywords.  The following is set up by default: 1) the room name is set to be the course name, 2) course ID and password are generated, 2) the room image is the default course room image, 3) other room and course configuration settings are set to defaults, (1) 4) a passageway is formed from the office of the primary instructor to the course room and vice versa, (2) 5) the course is put in the primary instructors list of favorite places (2) 6) the door to the course room is unlocked.  After creation the instructor will need to fill in the syllabus, course materials, and course work sections as appropriate, as well as enroll or invite shadow users, students, co-instructors, and teaching assistants.  For each meeting day and time between the course dates, a session is created in the course calendar.

6. Course Features

 

      The Tapped In Course Manager essentially provides mostly the same features as the Prometheus course management system but in the broader online setting of the Tapped In community at large.  Tapped In users can chat (with other users in the same room and taking advantage of chat support tools) and private message or e-mail any Tapped In user.  Most Tapped In rooms (including course rooms) have a discussion board where users can post messages in threaded discussions.  Passageways between rooms allow users to move more easily between rooms.  Group and course rooms allow file uploading and downloading into a file space for that room.  Many of these features are partially or fully present in Prometheus but will be re-implemented or adapted to Tapped In as a whole.

      The features remaining specific to course rooms are the syllabus, which provides all kinds of course information, providing course materials online, allowing for course work to be graded and submitted online, and course groups.  The Tapped In version of a course group will simply be an instance of Tapped In group with the addition of an explicit association of the group with a course.  See Settings for more information on course groups. (1) The details of course specific features can be found in the Course Room Interface Organization section.

      One feature of rooms that may not be specified elsewhere is the ability of the owner of the room, i.e. the primary instructor, or any other user with key privileges, to lock or unlock the door to the room. (2) If the door is locked, then no user without key privileges can enter the room, even if they are a member of the course.  Only a user with key privileges can unlock the door again. Any user currently in the room may still leave the room even if the door is locked.  The current state of the door is indicated in the Tapped In interface.  If the door is locked, the room is listed as unavailable in the directories.

7. Course Room Interface Organization

 

7.1. Prometheus

 

      Prometheus uses a relatively flexible edit form paradigm, which allows teachers to create and modify content.  Most fields can be renamed and fields left empty do not appear in the non-edit view. This basic paradigm is upheld in the interface that follows, although whereas Prometheus essentially allows the user to edit a very long page, listing all the fields, the interface specified here edits small portions of information in place. (1)  That is, on a page featuring three main blocks of information, the user can edit the second while the first and third still appear static.  The Prometheus course manager is divided into distinct modules.  Any module of the course manager can be disabled for any particular user in a course.  This level of flexibility is probably unnecessary and if disabling modules is allowed at all, it will be for all users in the course.

      Prometheus also provides tools for editing field content, including 1) importing a file (2) 2) equation creation and insertion (3) 3) creating an internal link to another course manager unit (e.g. a day in the calendar) (2) and 4) spell checking (3).  In Prometheus, all four tools are provided for every single text box.  Some alternatives to this approach are to provide the tools in some central location for use at any time or to pick and choose which tools are to be available for particular fields.

      Another detail worth noting is that Prometheus field contents are printed in the html as to allow html tags to be executed, which while convenient, could be a little problematic in regards to security.  It would be worthwhile to investigate how and to what degree Prometheus parses field contents. (2)

7.2. Integration into the Tapped In Interface

 

      The current concept of a Tapped In room currently consists of the content area within which is a sub-navigation bar for the room, and a detachable chat/private message window.  The sub-navigation bar is on the left side of the screen and has a picture of the room and a set of sub-navigational links. (1) A course room is a Tapped In room with added course managing functionality, i.e. with additional or slightly different sub-navigational links allowing the user access to course features. As mentioned above, course manager functionality is only active when the user is in the room for particular course (all courses must have a room).

      Some features that are mentioned in this document as part of the course manager would appear throughout Tapped In (chat, passageways, for example) while others are specific to course management.  Functionality specific to course management will always appear as links in the sub-navigation bar on the left side of the content area, rather than the interface at large. 

      The sub-navigation bar organizes the links in hierarchy.  Top level links are always visible.  Second level links are visible only for the top level page the user is currently viewing. (2) Any page that is three levels or deeper into the hierarchy of the page will never appear in the sub-navigation.  When the user is viewing a particular page, the link for that page in the sub-navigational bar will be disabled.

7.9.3. Teacher

 

      The settings page allows instructors and teaching assistants with settings edit privileges to modify the settings for the course.  These include the configuration of the room itself, configuration of course attributes, management of student course groups, and management of those enrolled in the class.

      The room configuration page allows the user to configure attributes common to all Tapped In rooms, such as the name of the room, the appearance of the welcome page, etc.  This is specified more completely in the Settings specification.

      The course configuration room allows the user to change course settings, such as the privacy setting, search keywords, or the password for the course.  The primary instructor can delete the course from this page.  The information page text for the course can also be edited here.  This is the text that appears on the course information page available to non-members of the course (see the Course Information Page section).  The welcome text can be used for the information page or different text can be specified.

      At the group management page, the user can create, edit, and delete student groups.  A student group is a normal Tapped In group with a simplified creation process and in which the instructor of the course has some owner-like privileges even if he or she is not the owner.  The initially chooses only the name and description of the group and can populate and remove students from the groups.  Student groups have the following features by default (2):

-         transcripts are on

-         group is private

-         tenant is the same as course tenant

-         group expires on course end date

-         room picture uses course room picture

In addition, a two passageway is automatically created between the course room and the group room. The group is also automatically placed in the favorite places of all members of the group. (2)  The instructor can choose to either make him/herself the owner of the group or not a member of the group at all. (2)  However, even if the instructor is not a member, they still have the ability to remove or add group members and can delete the group.  Course groups are explicitly linked to the course.  The teacher has the option of randomly assigning all students into the existing course groups in equal numbers or as close as possible. (3)

      The student management page provides a means to invite students, teaching assistants, and instructors to enroll in the course, drop any participant in the course, and configure edit privileges for teaching assistants.  New members of the course can either be invited to enroll (sent a private message) in the course or be automatically enrolled.  If a potential member is invited, he or she must respond to the invitation to be enrolled.  A list of pending enrollment invitations is listed on the page.  Automatically enrolling a member means that he or she receives a notification but does not need to accept an invitation to be a member of the course.  If the course is not private, a list of pending enrollment requests is also listed here where they can be accepted or rejected.

      Any member of the course can be changed to another user level here.  For example, a shadow user could be changed to a student and vice versa.  Any member of the course (except the primary instructor) can be dropped from the course here (who will receive notification via private message).  Teaching assistants can be given edit privileges to the various area of the course room here as well as a key to the room, allowing them to lock and unlock the door to the course room.

      Even if a teaching assistant has Student Management edit privileges, a teaching assistant can never have any effect on teaching assistant or instructor level users.  This means that a teaching assistant cannot invite, enroll, or accept/decline enrollment requests for other teaching assistants or instructor level users for this course.  A teaching assistant cannot remove or change the user level of instructors or fellow teaching assistant.  Teaching assistants cannot modify their own edit privileges or those of other teaching assistants.  Finally, teaching assistants cannot promote a student or shadow level user to teaching assistant or instructor level status.

      Students and shadow users never have access to any settings pages, instructors always do, and teaching assistant accounts can be allowed settings access or not.

Course Building and Management

 

·         Course Content Creation. Prometheus organizes course content around a syllabus model. Using a self-guided, question-and-answer format, instructors move easily through content creation. Guides within Prometheus are mere suggestions. Instructors can place content in suggested fields or disregard them. Any disregarded fields will not be visible to students in the syllabus. Because the content creation model in the syllabus section is consistent throughout Prometheus, instructors who have reached a comfort level there can easily create content in any section. The Prometheus course creation model works like this. The instructor

·         titles the content;

·         enters new content or imports existing content;

·         uploads text, multimedia, PowerPoint, or other files to enhance content;

·         associates notes with content files.

 

·         Modular Organization, Prometheus provides instructors with the ability to create learning objects outside the course context, group them into "content modules," then insert and rearrange them within courses. Such modules might contain outline items, assignments, tests, projects, lectures, files, and discussion topics.

 

·         Modular Content Distribution, Modular learning objects make content independent of any particular course, allowing reuse in multiple courses. Changes made to a module replicate across courses, enabling centralized content management. Instructors can reassign which modules are associated with which courses using a simple drag-and-drop interface.

 

·         The Prometheus summer 2001 release will allow distribution of entire course "cartridges.

 

Testing and Grade Book

 

  • Prometheus allows for quantitative assessments using multiple-choice or other objective testing formats as well as open content assessment, such as short answer and essay formats.

 

  • The testing feature allows online administration of tests and surveys. From the testing section, instructors can create and edit tests and surveys, grade tests, and view results.

 

  • Prometheus can be set to post online tests and corresponding grades automatically to a grade book. The grade book lists all students registered for a course. Scores edited in the testing section will automatically be updated in the grade book. Instructors can grant students permission to see only their own grades, anonymous grades for the entire class, or the grade book can be closed to student view entirely.

 

  • Instructors can post grades by student or by item. They can also add comments, reset, edit, or delete the entire grade book or specific items within the grade book. Instructors can email a student by simply clicking his or her name in the grade book.

 

  • Instructors can also download their grade book to Microsoft Excel by clicking one button.
