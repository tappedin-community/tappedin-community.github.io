Research Tools

We need to have all data collected and stored in a database. We need to talk about the design of the database. A supporting document called Database fields lists all the current fields in the current TI database, plus additional ones that we need. The database needs to be designed in a way that makes more sense to the researchers than the current TI's database. We will need discuss the design. I don't know the best way to design databases, but I can tell you what has not worked for me and why. If possible the database should be on a different server than the one the community is on. The database should collect all commands issued from both the web interface and the chat interface. The database should also have demographic information about the members.

We will need Admin Reporting tools and Researcher Reporting tools. The distinction will be that Admin Reporting tools will provide summary data and Researcher Reporting tools will provide raw data.

We also need a way to look at guests. We won't be able to assume anything across guest log ins, but we will need a way to count guest log ins, and to follow what a guest does with in a log in session so that we can learn something about guests.

Admin Reporting Tools

All of the repoting tools should have the ability to be run by any time frame. Another dimension that is independent from time frame is that reporting tools should be able to be run by any location group, room, course, or all of TAPPED IN. For time frame we should be able to break things down by session, hour, day, week, month or other specified duration. Session is defined in the Calendar spec. A session has the following fields associated with it and we should be able to do searches with these fields too:

  • Date of session
  • Start time and approximateend
  • Room
  • Session name
  • Description
  • Leader name
  • keyword categorization (so people can search and we can sort by keyword)
  • Group affiliation

We will need summary data of people who attended during a period of time (session, hour, day, week, month, any specified duration). For example, SSF sessions that were listed on the SSF calendar for June. The report should have two options, list everyone who attended the session and not just the members of the SSF group (assuming they are a public group and anyone can attend their sessions), or just list group members who attended the session. We also need to look at just TI members or include guest data too. By default, summary tools should only provide info about a group's members to group admin, but it could be an option for them.

Researcher tools:

Many of the Admin tools will also function as researcher tools, but in addition, researchers will need some special capabilities

In general, content is not collected for research purposes--transcripts of public utterances are captured for public ASO sessions and a group decides whether to archive their own discussions--but under some circumstances it is necessary to collect all utterances (public and private) with the user's permission for research. Also collect ALL commands issued during a certain session (synchronous and asynchronous commands and the order in which they occurred and time stamped.)' This would be a complete log of everything that occurred in the room during the discussion. For example if a user spoke and then clicked on the discussion board all this would be recorded as well as the content of what was spoken and what was read or written to the discussion board.
